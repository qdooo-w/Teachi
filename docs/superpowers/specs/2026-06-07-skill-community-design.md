# 技能社区与仓库系统界面及交互设计规范

## 1. 背景与目标

为了替换原有扁平化的技能管理方式（原 `SkillManagerDialog` 弹窗与旧 `CommunityView` 极简 ZIP 上传），系统全新演进为“运行层 (Runtime) -> 仓库 (Library) -> 社区 (Community)”三层隔离架构。本规范致力于定义所有前端界面和交互流程细节，确保：
1. 用户私有技能的编辑、管理和从本地收集（Import）流程顺畅、安全。
2. 公开社区的浏览、详情展示、评论互动与排行榜具有高信息浓度与卓越的视觉表现力。
3. 两级审核流（技能管理员审核、社区全局管理员审核）有独立的管理界面支撑。
4. 严格遵守原有 Learnova 技能文件系统规范与校验规则（如 Kebab-case 命名规则、Frontmatter 约束、支持的文件夹与文本/图片类型等）。

---

## 2. 交互组织与路由规划

系统全面采用 **全路由页面模式**。隐藏原有右侧顶部 Header 中的 "Skills" 弹窗按钮，将主入口统一收纳至侧边栏左下角：

```
侧边栏底端 (Sidebar Bottom)
├── 科目总览 (现有)
├── 个人中心 / 设置 (现有)
├── 💼 我的技能仓库 (新增，导航至 /library)
└── 🌐 公开技能社区 (新增，导航至 /community)
```

### 路由表设计：
* `/library` — 技能仓库主页（卡片网格列表）
* `/library/edit/:id` — 技能文件编辑工作区（文件树 + 专属编辑器）
* `/library/collect` — 运行层至仓库的导入表单页
* `/library/publish/:id` — 仓库到社区的发布/更新表单页
* `/community` — 社区主页（含列表、排行榜、审核控制台 Tab 切换）
* `/community/skills/:id` — 社区技能详情页（README 渲染 + 评论互动 + 贡献者列表）

## 3. 页面界面设计规范

### 3.1 技能仓库主页 (`/library`)
* **视觉结构**：卡片网格（Grid of Cards），设计语言符合 Learnova 主卡片风格。
* **卡片展示信息**：
  * 技能展示名 (`display_name`) 与唯一标识名 (`name`)。
  * 简短描述 (`description`)，超出 2 行使用 CSS line-clamp 截断。
  * 分类标签行 (`tags`)。
  * 版本号 (`version`) 标识。
* **交互操作**：
  * 点击任意卡片，无缝路由跳转至 `/library/edit/:id` 进行编辑。
  * 顶部常驻 `+ 收集本地技能` 按钮，点击跳转至 `/library/collect`。
  * 支持按 `tags` 过滤和拼音/汉字关键词检索。
* **审核 Tab**（仅在用户是某些技能的管理员 `admin_uuids` 时显示）：
  * 提供 `贡献审核` 子板块，展示来自其他贡献者提交给该技能的新版本（`status = PENDING_OWNER`）。
  * 技能管理员可在此一键批准（流转为 `PENDING_ADMIN`）或驳回（强制填写 `reject_note`，写回 `skill_review_logs`）。

### 3.2 技能编辑工作区 (`/library/edit/:id`)
* **视觉结构**：左右分栏的双栏沉浸式编辑器。
* **左侧文件树 (File Tree)**：
  * 展示当前技能包下的所有物理文件（如 `SKILL.md`，以及 `references/`, `assets/` 下的子文件）。
  * 支持右键或顶部小按钮新增文件/新建资源目录（校验合法性）。
* **右侧编辑器 (Editor Zone)**：
  * 文件加载 loading 指示。
  * 针对 `SKILL.md` 提供 YAML Frontmatter 表单预览与 prompt 文本框编辑器；针对 `.md` 提供带 Preview 的 Markdown 编辑器；针对 `.json`, `.yaml` 等提供语法高亮编辑器。
  * 底部常驻 `保存修改` 按钮。
  * **原技能规则校验约束**：
    * 保存时，前端调用 `validatePlainSkillFileContent` 校验。
    * 如果是 `SKILL.md`，保存前必须调用 `parseSkillFrontmatter`，确保 `name` 和 `description` 不为空、YAML 结构合法，否则拦截保存并展示错误横幅。
    * 仅允许创建合法的子文件夹（`references`, `assets`, `examples`, `templates`）和合法的文件类型（文本 `.md/.txt/.json/.yaml/.yml` 或图片格式）。

### 3.3 公开技能社区 (`/community`)
* **视觉结构**：纵向高信息浓度的表格行列表（Vertical rows），告别低效卡片平铺。
* **列表项行展示信息**：
  * **第一列 (标题描述)**：技能友好名称 + 括号内的唯一标识名 (kebab-case)，下方紧跟简短描述。
  * **第二列 (Tags)**：圆角小标签集合。
  * **第三列 (版本)**：当前最新 Approved 版本号（等宽字体展示）。
  * **第四列 (数据统计)**：累计下载量 (带有 `⬇` 图标) 与点赞数 (带有 `❤️` 图标)。
  * **第五列 (作者与日期)**：作者用户名与最后更新日期。
* **交互操作**：
  * 每一行悬浮时背景微变色，点击直接路由跳转至 `/community/skills/:id`。
  * 顶部常驻关键字全局搜索框，与标签多选筛选器。
* **社区排行榜 Tab**：
  * 展示三个独立的榜单：`热门下载榜`、`最具人气点赞榜`、`金牌贡献者榜`。
  * 榜单项左侧显示醒目的金银铜牌或名次角标。
* **全局审核控制台 Tab**（仅对 `users.role == 'admin'` 的全局管理员可见）：
  * **上架版本审批**：列出 `PENDING_ADMIN` 状态的申请，管理员可查看源码包，并点击 `批准上架` (APPROVED) 或 `驳回` (REJECTED_ADMIN)。
  * **敏感评论处理**：列出有用户举报的评论 (PENDING 状态)，管理员可查看原文并“一键驳回”或“删除评论”。

### 3.4 社区技能详情页 (`/community/skills/:id`)
* **视觉结构**：左侧 README 主展示区 + 右侧侧边栏信息卡片。
* **左侧 README 区**：
  * 将最新版本的 `readme_md` 内容（同级 `README.md` 文件内容）通过 `markdown-it` 渲染为精美的富文本页，保持与聊天消息渲染器风格一致。
* **右侧侧栏**：
  * 核心按钮组：
    1. `安装到我的仓库`：将此公开版本继承安装到用户私有仓库。
    2. `安装到特定项目运行层`：选择用户已有的项目（Dropdown），将文件纯复制到项目目录，提升 `downloads` 计数。
  * 详细指标面板：当前版本号、文件大小、累计下载量、点赞状态（可 Toggle 触发 `/like` API 并刷新界面上的 ❤️ 计数）。
  * 贡献团队模块：展示创建者 (Owner) 以及贡献者 (Contributors) 列表。
* **下方评论区**：
  * 输入框：支持 Markdown 发表评论。
  * 评论树：最大嵌套深度限制为 1（两层结构）。
    * 第一层评论（depth=0）：展示内容，右侧带点赞按钮与“回复”键。
    * 第二层评论（depth=1）：缩进排版，显示“回复 @username”。对二级评论进行回复时，仍作为同级回复排列在下方，但 `reply_to_uuid` 会指向具体被回复的用户。
    * 举报（Report）：每条评论均可点击“举报”，弹出举报原因表单提交至后台。

## 4. 数据流与细节交互设计

### 4.1 导入收集表单页 (`/library/collect`)
1. **触发方式**：在会话页面中或项目设置中选择“导入此本地技能”，路由重定向至 `/library/collect?scope=xxx&pid=xxx&skill_name=xxx`。
2. **预填与模板匹配逻辑**：
   * 前端向 `/library/skills/parse-runtime` 发送请求。
   * 后端读取本地 `SKILL.md`，返回 parsed frontmatter 信息。
   * 后端通过 `name` 自动匹配仓库里最新的同名技能，返回 `matched_template`。
   * 如果匹配成功，表单自动套用模板中已有的 `display_name`, `tags`, `readme_md` 字段。
   * 自动递增版本号：如模板版本为 `1.2.0`，前台自动推荐 `1.3.0` 或 `1.2.1`。
   * 用户亦可在模板下拉选择框中手动指定仓库中的任意其他技能作为模板，或选择“全新导入”。
3. **用户填写**：确认显示名、标签、手动输入**本次导入版本号**与**更新说明 (Changelog)**。
4. **提交行为**：提交后，后台生成新 `user_library_skills` 记录与独立 UUID，物理复制文件夹，最后路由跳转回 `/library` 主页。

### 4.2 社区发布与更新表单页 (`/library/publish/:id`)
1. **触发方式**：在 `/library` 列表或编辑页点击“发布到社区”，跳转至 `/library/publish/:id`。
2. **数据绑定**：
   * 获取该仓库技能的数据。名称、版本号、标签、大小、README 为静态只读（与仓库保持一致）。
   * 自动判定目的地：如果该技能已在社区主表中，且当前用户有权限，自动判定为“追加版本并提交审核 (PENDING_ADMIN 或 PENDING_OWNER)”；否则，显示“首次发布该技能”。
3. **用户填写**：本次版本的 **Changelog (必填)** 以及 **Fork 来源说明 (可选)**。
4. **提交行为**：提交后后端写表并打包物理文件，页面重定向回 `/library` 或 `/community`。

## 5. 错误处理与健壮性设计

1. **版本号规范约束**：导入与发布时，版本号必须符合语义化版本格式（Semantic Versioning, e.g. `X.Y.Z`），前端输入框做正则防错校验。
2. **重名冲突处理**：
   * 仓库：允许同名多个卡片存在（因为没有 `UNIQUE(user_uuid, name)` 约束，用不同 `id` 和版本区分）。
   * 社区：`community_skills` 主表的 `name` 字段为 `UNIQUE`。首次发布技能时，如果社区中已存在相同 `name`（且不是用户自己所拥有），系统必须抛出“该技能标识已存在，如为改进版请尝试申请为贡献者”的警告。
3. **无障碍设计与焦点**：
   * 表单编辑区在输入非法字符（如技能标识包含非法下划线时）立即触发红色警示，禁用提交按钮。
   * 评论框提交时，防抖（Debounce）处理，避免快速重复点击生成多条评论。

## 6. 测试策略与验证计划

1. **UI 单元测试 (`vitest` + Vue Test Utils)**：
   * 编写前端单元测试，验证 `SKILL.md` 解析与 `parseSkillFrontmatter` 规则的匹配情况。
   * 测试 `validateSkillName` 对中英文、连字符的边界检测。
2. **API 连通性集成测试 (`pytest`)**：
   * 在后端 `tests/` 下建立测试集，覆盖 `/library/skills/collect` 对数据继承、文件物理复制的实现。
   * 模拟普通贡献者提交审核至 `PENDING_OWNER`，管理员批准转至 `PENDING_ADMIN`，全局管理员批准完成 `APPROVED` 上架的完整生命周期流转。
